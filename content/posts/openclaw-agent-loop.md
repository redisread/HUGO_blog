在构建 AI 应用时，我们往往关注模型的能力和提示工程，却忽略了"智能体循环"这个核心机制。OpenClaw 作为一个开源的 AI 智能体平台，其智能体循环设计代表了当前业界最先进的实现思路。

本文将深入解析 OpenClaw 的智能体循环，带你了解从接收消息到生成回复的完整链路。

<!--more-->

## 什么是智能体循环？

智能体循环（Agent Loop）是智能体的完整运行周期：**接收 → 上下文组装 → 模型推理 → 工具执行 → 流式回复 → 持久化**。这是将消息转化为操作和最终回复的权威路径，同时保持会话状态的一致性。

简单来说，它是 AI 智能体的"心跳"——每一次用户交互都要经过这个循环的完整处理。

## OpenClaw 智能体循环的 5 个阶段

### 阶段 1：入口与验证

当用户发送消息时，循环从以下入口点开始：

- **Gateway RPC**：`agent` 和 `agent.wait`
- **CLI**：`agent` 命令

此时系统会：
1. 验证参数合法性
2. 解析会话（sessionKey/sessionId）
3. 持久化会话元数据
4. 立即返回 `{ runId, acceptedAt }` 确认接收

**关键设计**：采用"先确认后处理"模式，确保用户得到即时反馈，实际处理在后台异步进行。

### 阶段 2：会话准备与环境构建

在模型开始推理之前，OpenClaw 需要准备运行环境：

**工作区准备**：
- 解析并创建工作区
- 沙箱隔离运行可能会重定向到沙箱工作区根目录
- 获取会话写锁，确保数据一致性

**Skills 加载**：
- 加载 Skills 快照（或从缓存复用）
- 将 Skills 注入到环境和提示中
- 解析引导/上下文文件并注入系统提示

**提示组装**：
- 基础提示 + Skills 提示 + 引导上下文 + 运行覆盖
- 强制执行模型特定的限制和压缩保留令牌

### 阶段 3：模型推理与流式传输

这是循环的核心阶段，由 `runEmbeddedPiAgent` 函数驱动：

**队列与并发控制**：
- 运行按会话键（session 通道）序列化
- 可选择通过全局通道控制并发
- 防止工具/会话竞争，保持会话历史一致性

**推理过程**：
1. 解析模型 + 认证配置文件
2. 构建 pi 会话（OpenClaw 的底层运行时）
3. 订阅 pi 事件并流式传输增量
4. 强制执行超时（默认 48 小时，可配置）

**流式事件类型**：
- `stream: "assistant"` —— 助手增量（文字回复）
- `stream: "tool"` —— 工具调用事件
- `stream: "lifecycle"` —— 生命周期事件（start/end/error）

### 阶段 4：工具执行与结果处理

当模型决定调用工具时，循环进入工具执行阶段：

**工具生命周期**：
1. `tool_start` —— 开始执行
2. `tool_update` —— 进度更新（适用于长时任务）
3. `tool_end` —— 执行完成，返回结果

**结果处理**：
- 清理工具结果（大小限制、图像负载处理）
- 跟踪消息工具发送，防止重复确认
- 将结果注入回上下文，供下一轮推理使用

### 阶段 5：回复整形与持久化

最后阶段将处理结果转化为最终输出：

**回复组装**：
- 助手文本（和可选的推理过程）
- 内联工具摘要（详细模式）
- 错误处理（模型出错时的回退文本）

**输出过滤**：
- `NO_REPLY` 静默令牌过滤
- 消息工具重复项移除
- 工具错误回退（除非已发送用户可见回复）

**持久化**：
- 将完整会话历史写入存储
- 更新会话元数据
- 释放会话锁

## 核心机制详解

### 队列与并发控制

OpenClaw 采用**单会话序列化**设计：

```
会话 A：消息 1 → 处理 → 消息 2 → 处理 → 消息 3 → 处理
会话 B：消息 1 → 处理 → 消息 2 → 处理（与 A 并行）
```

这种设计的优势：
- **避免竞态条件**：同一会话的消息按顺序处理
- **保持上下文一致性**：历史记录不会乱序
- **支持全局并发**：不同会话可并行处理

消息渠道可选择队列模式（collect/steer/followup）来馈送此通道系统。

### 钩子系统：可扩展的拦截点

OpenClaw 提供两套钩子系统，允许开发者在关键节点插入自定义逻辑：

**内部钩子（Gateway 层）**：
- `agent:bootstrap` —— 在系统提示最终确定前修改引导文件
- 命令钩子 —— 拦截 `/new`、`/reset`、`/stop` 等命令

**插件钩子（智能体生命周期）**：
- `before_agent_start` —— 注入上下文或覆盖系统提示
- `before_tool_call` / `after_tool_call` —— 拦截工具参数/结果
- `tool_result_persist` —— 转换工具结果后再写入会话
- `message_received` / `message_sending` —— 入站/出站消息处理

这种设计让 OpenClaw 既保持核心简洁，又支持强大的扩展能力。

### 压缩与重试机制

当会话历史过长时，OpenClaw 会触发**自动压缩**：

1. 发出 `compaction` 流事件
2. 重置内存缓冲区和工具摘要
3. 避免重复输出，触发重试

压缩策略可配置，支持多种压缩算法和保留策略。

### 超时与取消

OpenClaw 提供多层超时保护：

| 层级 | 默认超时 | 说明 |
|------|----------|------|
| `agent.wait` | 30 秒 | 仅等待运行完成 |
| 智能体运行时 | 48 小时 | 强制中止长时间运行 |
| 工具调用 | 可配置 | 单个工具执行超时 |

支持通过 `AbortSignal` 主动取消运行。

## 实际应用场景

### 场景 1：多轮工具调用

用户请求："帮我查一下北京的天气，然后发邮件提醒我带伞"

循环执行：
1. 接收消息，组装上下文
2. 模型推理 → 调用天气工具
3. 等待工具结果 → 注入上下文
4. 模型推理 → 调用邮件发送工具
5. 等待结果 → 生成最终回复
6. 持久化完整会话

### 场景 2：长时间任务

用户请求："分析这个 100MB 的日志文件"

循环特点：
- 工具执行阶段可能持续数分钟
- 通过 `tool_update` 发送进度更新
- 用户可实时看到处理进度
- 超时保护防止无限等待

### 场景 3：并发会话

用户 A 和 B 同时发送消息：

- 会话 A 和 B 的处理**并行进行**
- 各自的消息在会话内**串行处理**
- 互不影响，资源隔离

## 最佳实践建议

### 1. 合理配置超时

```yaml
# openclaw.yaml
agents:
  defaults:
    timeoutSeconds: 3600  # 1小时，适合大多数场景
```

### 2. 利用钩子扩展功能

```javascript
// 在工具调用前添加日志
hooks.before_tool_call = async (ctx, toolCall) => {
  console.log(`[${ctx.sessionId}] 调用工具: ${toolCall.name}`);
  return toolCall;
};
```

### 3. 监控生命周期事件

```javascript
// 监听运行状态
agent.on('lifecycle', (event) => {
  if (event.phase === 'error') {
    alert('运行出错: ' + event.error);
  }
});
```

### 4. 优化提示组装

- 精简 Skills 提示，避免token浪费
- 使用引导文件注入常用上下文
- 定期清理历史记录，触发压缩

## 总结

OpenClaw 的智能体循环设计体现了以下核心理念：

1. **异步确认**：先响应，后处理，提升用户体验
2. **会话隔离**：单会话串行，多会话并行，保证一致性
3. **流式传输**：实时反馈，进度可见
4. **可扩展性**：钩子系统支持深度定制
5. **健壮性**：超时保护、错误处理、自动压缩

理解这个循环机制，能帮助你更好地：
- 调试智能体行为
- 优化响应延迟
- 设计复杂的 Agent 工作流
- 扩展 OpenClaw 功能

智能体循环是 AI 应用的"操作系统"，掌握它是构建可靠 Agent 系统的关键一步。
